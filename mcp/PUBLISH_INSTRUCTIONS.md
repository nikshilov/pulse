# Publishing `@nikshilov/pulse-mcp` to npm

Step-by-step for first publish (and subsequent releases). Niki executes manually — agents do prep + dry-run only.

## Pre-flight checklist (run before every publish)

- [ ] Local Pulse engine on `127.0.0.1:18789` is up and `/retrieve` answers a smoke query (`curl -X POST http://127.0.0.1:18789/retrieve -H "Content-Type: application/json" -d '{"query":"hello","top_k":1}'`).
- [ ] `version` in `package.json` is bumped (semver). For first publish: `0.1.0`. Subsequent: `0.1.1` (patch), `0.2.0` (minor breaking-tools), etc.
- [ ] `README.md` reflects current tools and env vars (no stale references).
- [ ] `git status` clean in `pulse/` repo (or at least `mcp/` subdir).
- [ ] `npm run build` succeeds.
- [ ] `npm publish --dry-run` shows ≤10 files, ≤50 KB tarball, no `node_modules`, no `src/`, no `*.tsbuildinfo`.

## Publish

```bash
cd ~/dev/ai/Garden/pulse/mcp
npm publish
```

That's it. `publishConfig.access: "public"` in `package.json` already handles the scoped-package public flag — no need to pass `--access public` manually (though it's harmless).

Expected output: `+ @nikshilov/pulse-mcp@0.1.0`

Verify: `npm view @nikshilov/pulse-mcp` → should show all metadata.

Live install test from a fresh terminal:

```bash
npx -y @nikshilov/pulse-mcp
# should print: [pulse-mcp v0.1.0] connected via stdio; backing Pulse: http://127.0.0.1:18789
# (Ctrl-C to exit — it's an MCP stdio server, expects MCP client framing)
```

## Bumping versions

```bash
cd ~/dev/ai/Garden/pulse/mcp

# Patch (bug fix, e.g. 0.1.0 → 0.1.1):
npm version patch

# Minor (new tool, backward-compat, e.g. 0.1.0 → 0.2.0):
npm version minor

# Major (breaking change, e.g. 0.1.0 → 1.0.0):
npm version major
```

`npm version` also creates a git tag. Then publish:

```bash
npm publish
git push --tags
```

## Rollback / "I shipped a broken version"

**npm does NOT allow `unpublish` after 72h** (and even within 72h, only if no one has installed the package).

Correct rollback path:

1. **Bump and publish a fixed version immediately.** E.g. if `0.1.0` is broken, fix the bug, bump to `0.1.1`, publish.

2. **Deprecate the broken version** so users see a warning when installing it:

   ```bash
   npm deprecate @nikshilov/pulse-mcp@0.1.0 "Broken: pulse_recall returns 500. Use 0.1.1+."
   ```

3. **Update README** to mention the minimum supported version if needed.

If the broken version was published <72h ago AND no one installed it, you _can_ try:

```bash
npm unpublish @nikshilov/pulse-mcp@0.1.0
```

…but plan around the assumption you can't unpublish.

## Common gotchas

- **`npm ERR! 403 Forbidden`** — usually means scope mismatch (publishing `@nikshilov/foo` while logged in as a different user). Run `npm whoami` to confirm.
- **`npm ERR! E402 Payment Required`** — scoped packages without `publishConfig.access: public` default to private (paid). `package.json` already has `publishConfig.access: public` — don't remove it.
- **`prepublishOnly` script doesn't run** on `npm publish --dry-run` in some npm versions — this is OK; the build script runs explicitly.
- **`npm view @nikshilov/pulse-mcp` returns 404** right after publish — npm registry CDN takes a few seconds to propagate. Wait 30s and retry.

## File ownership

- `mcp/package.json` — version + metadata (Niki edits version manually).
- `mcp/README.md` — public-facing docs (update with each new tool / breaking change).
- `mcp/src/index.ts` — MCP wrapper code; touched when Pulse server adds new endpoints.
- `mcp/dist/` — generated by `npm run build`; gitignored; never edited by hand.
- `mcp/PUBLISH_INSTRUCTIONS.md` — this file. Update when publish flow changes.
